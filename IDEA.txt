# SentinelAI — Project Brief

## What it is

SentinelAI is a self-hosted AI gateway and observability platform. It sits between
applications and LLMs (OpenAI, Anthropic, Groq, Mistral, Cerebras) and manages all
AI traffic in a controlled, secure, and measurable way.

Instead of every app calling LLM APIs directly with no guardrails, no logging, and no
cost tracking, all requests go through this platform, which:

- Routes requests to the right model and provider
- Enforces security checks (prompt injection, PII redaction, content policy)
- Retrieves enterprise knowledge when needed (RAG with hybrid search)
- Runs agent workflows with real tool calls
- Tracks every request: latency, tokens, cost, errors, traces
- Evaluates response quality asynchronously (faithfulness, relevance, coherence)
- Stores immutable audit logs for compliance


## Why this project

The target is backend + AI infra roles. This project avoids the "another chatbot wrapper"
problem by being a platform with real production concerns:

- Distributed workers and queues (BullMQ)
- Multi-tenant auth with SHA-256 hashed API keys
- PostgreSQL schema with pgvector for embeddings and semantic search
- Redis for rate limiting and caching
- OpenTelemetry-style trace spans
- Prometheus metrics and Grafana dashboards
- Docker Compose + Nginx + GitHub Actions CI/CD on a live DigitalOcean droplet

An interviewer can spend 20 minutes on the architecture alone.


## Inspiration

Three open-source projects shaped the design. None were copied — their ideas were
studied and then re-implemented from scratch in a unified backend.

| Source        | What was borrowed                                           | Why                                                         |
|---------------|-------------------------------------------------------------|-------------------------------------------------------------|
| Helicone      | Unified provider abstraction, routing, cost/usage tracking  | One gateway for many models instead of direct API sprawl    |
| ArchGW        | Proxy-style prompt guardrails, resilient routing, tracing   | Treat AI traffic like network traffic needing a control plane |
| OpenLLMetry   | OTel-style spans, metrics, trace-based debugging            | LLM systems need the same observability as any backend service |

The interview framing: "I identified three production problems — provider sprawl, missing
guardrails, and poor observability — then studied how the industry solves each one and
built a unified backend platform around those patterns."


## Architecture

```
npm workspace monorepo

shared/                    — TypeScript types + queue name constants
services/
  gateway/                 — Fastify HTTP server, port 3000
  ingestion-worker/        — BullMQ worker: chunk + embed documents
  eval-worker/             — BullMQ worker: score LLM response quality
infra/
  postgres/init.sql        — Full DB schema with pgvector
  nginx/nginx.conf         — Reverse proxy
  prometheus/              — Metrics scrape config + alert rules
  grafana/                 — Dashboard provisioning
docker-compose.yml         — Full local stack
```

Request path for POST /v1/chat:

```
auth plugin (SHA-256 key lookup)
  → rate-limit plugin (Redis, per-tenant)
  → guardrails (injection detection, PII redaction)
  → budget check (monthly spend limit, 402 if exceeded)
  → semantic cache check (pgvector cosine ≥ 0.95, skip if session/RAG/stream)
  → router (picks provider/model, A/B experiment override)
  → session memory load (conversation history + context guard)
  → RAG retrieval (hybrid vector + keyword search, RRF reranking)
  → LLM call with retry + exponential backoff
  → persist llm_requests row
  → async: flush trace_spans
  → async: enqueue eval job
  → async: save session
  → response with session_id, turn count, ab_variant, cost
```


## Features built

**Core gateway**
- Multi-provider LLM routing: OpenAI, Anthropic, Groq, Mistral, Cerebras
- Automatic fallback to backup provider on failure
- Exponential backoff retry (100ms → 2s, ±50ms jitter, 3 attempts)
- Per-tenant API key auth (SHA-256 hash, scopes, rate limits)
- Redis-based rate limiting per tenant (RPM)

**Security**
- Prompt injection detection (regex patterns)
- PII redaction before LLM call
- Immutable audit log (append-only, no UPDATE/DELETE)
- Monthly cost budget enforcement with webhook alerts

**RAG**
- Document upload, chunking, Mistral embedding (vector(1024))
- Hybrid search: pgvector cosine + PostgreSQL tsvector full-text
- Reciprocal Rank Fusion reranking (k=60)
- Citation attachment in responses

**Semantic cache**
- pgvector cosine similarity threshold (≥ 0.95)
- 24-hour TTL, per-tenant, skipped for sessions and streaming

**Conversation memory**
- Server-side session storage (conversation_sessions table)
- Context window guard: auto-summarizes old turns at 75% of model limit
- Keeps last 4 turns fresh, injects summary as system message
- Session list, get, and delete endpoints

**A/B experiments**
- Probabilistic traffic split (configurable %)
- Control vs variant provider/model
- Experiment ID and variant tagged in every response

**Agent runtime**
- ReAct-style tool-calling loop (web search, calculator, knowledge base)
- Streaming mode via SSE (per-step events)

**Observability**
- OTel-style trace spans per request
- Prometheus metrics: request count, latency histogram, token usage, cost
- Grafana dashboard
- Alert rules: high error rate, high latency, guardrail trigger rate, service down
- Eval worker: faithfulness, relevance, coherence scores per response

**Admin API**
- API key management (create, list, revoke)
- Budget management (upsert, status)
- A/B experiment management (create, list, update, stop)
- Semantic cache stats and flush
- Audit log query with action filter

## Database schema (key tables)

- tenants, api_keys — multi-tenant auth
- llm_requests — one row per request, full audit trail
- trace_spans — OTel-style spans linked by trace_id
- document_chunks — chunked text + vector(1024) + tsvector for hybrid search
- eval_results — per-request quality scores
- audit_logs — append-only, no UPDATE/DELETE
- conversation_sessions — server-side message history + summary
- semantic_cache — query embedding + response, TTL-based
- tenant_budgets — monthly spend limits
- ab_experiments — A/B routing experiment config

## Interview talking points

- Why a gateway instead of direct API calls?
  Centralizes auth, rate limiting, cost tracking, guardrails, and observability. Every
  app gets these for free without implementing them individually.

- How does the semantic cache work?
  Each query is embedded via Mistral. On new requests, we run a cosine similarity query
  in pgvector against cached embeddings. If the nearest match exceeds 0.95, we return
  the cached response without calling the LLM. TTL is 24 hours.

- How does hybrid search differ from pure vector search?
  Vector search finds semantically similar chunks but misses exact keyword matches.
  Full-text search (tsvector) is precise for exact terms but blind to semantics.
  Reciprocal Rank Fusion merges both ranked lists: score = sum(1 / (60 + rank))
  across both result sets, giving documents a boost if they rank well in either.

- How does the context guard work?
  When a session exceeds 75% of the model's token limit, we take all turns except the
  last 4, summarize them into 2-4 sentences using a fast cheap model (llama-3.1-8b-instant),
  then store only the summary + fresh turns. The summary is injected as a system message.

- What happens if the primary LLM fails?
  withRetry() wraps every provider call: 3 attempts with exponential backoff (100ms,
  200ms, 400ms ± 50ms jitter). If all retries fail, getFallbackRoute() picks a secondary
  provider/model and retries once more.
